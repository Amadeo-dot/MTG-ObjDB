En  el  context  de  les  Bases  de  Dades  Orientades  a  Objectes  (BDOO)  i  JPA,  la  Inheritance  (Herència)  és  la  capacitat  de  traslladar  l'estructura  jeràrquica  de  les  teves  classes  Java  directament  a  la  base  de  dades.  
És  un  dels  punts  on  les  BDOO  guanyen  per  golejada  a  les  bases  de  dades  relacionals  (SQL),  ja  que  en  SQL  l'herència  no  existeix  de  forma  nativa  i  s'ha  de  "simular"  amb  taules  estranyes.  
Aquí  tens  què  significa  i  com  funciona  segons  els  teus  apunts:  
1.  El  concepte:  "És  un"  
L'herència  permet  definir  una  classe  genèrica  (Pare)  i  classes  més  específiques  (Filles)  que  hereten  els  seus  atributs.  
●  Exemple  Magic:  Una  Criatura és  una  Carta.  Una  Terra és  una  Carta.  ●  Totes  les  cartes  tenen  un  nom i  un  cost,  però  només  les  criatures  tenen  força i  resistència.  
2.  Els  grans  avantatges  en  BDOO  (ObjectDB)  
 
L'herència  permet  dues  coses  clau:  
●  Polimorfisme  en  les  consultes:  Pots  fer  una  consulta  a  la  classe  pare  i  la  base  de  dades  et  retornarà  


===== PAGE 6 =====
tots  els  fills.  ○  Consulta:  SELECT  c  FROM  Carta  c  ○  Resultat:  ObjectDB  et  retornarà  una  llista  on  hi  haurà  barrejades  Criatures,  Terres i  Encanteris.  Java  sabrà  de  quin  tipus  és  cadascuna  automàticament.  ●  Simplificació  del  model:  No  necessites  fer  un  "JOIN"  entre  la  taula  de  cartes  i  la  de  criatures.  Per  a  tu,  com  a  programador,  un  objecte  Criatura és  un  sol  bloc  que  ja  conté  tota  la  informació  de  la  Carta.  
3.  Estratègies  d'Herència.  
Encara  que  ObjectDB  ho  gestiona  de  forma  molt  eficient,  en  l'estàndard  JPA  existeixen  tres  maneres  de  guardar  l'herència:  
1.  Single  Table  (Una  sola  taula):  Tots  els  fills  van  a  la  mateixa  taula  (molts  camps  queden  buits/nulls).  És  la  més  ràpida.  2.  Joined  (Taules  unides):  El  pare  té  una  taula  i  cada  fill  una  altra.  És  la  més  neta.  3.  Table  per  Class:  Cada  classe  filla  té  la  seva  pròpia  taula  amb  tots  els  camps.  
En  resum:  La  inheritance  significa  que  la  base  de  dades  respecta  la  teva  estructura  de  classes  Java,  permetent-te  guardar  i  recuperar  objectes  complexos  mantenint  la  seva  relació  de  "pare  i  fill"  sense  haver  de  fer  servir  claus  foranes  manuals  per  connectar-los.  
4.1.  SINGLE_TABLE  (Una  sola  taula)  
Concepte:  Tot  es  guarda  a  la  taula  CARTA.  JPA  afegeix  una  columna  extra  "invisible"  per  saber  de  quin  tipus  és  cada  fila  (columna  discriminadora).  
És  l'estratègia  per  defecte  de  JPA.  Imagineu  que  tenim  la  classe  Carta i  les  filles  Criatura i  Terra.  
●  Com  es  guarda:  Es  crea  una  única  taula  anomenada  CARTA.  Aquesta  taula  té  totes  les  columnes:  les  del  pare  (nom,  cost)  i  les  de  tots  els  fills  (força,  color_produccio).  ●  Columna  Discriminadora:  Es  crea  una  columna  especial  (normalment  anomenada  DTYPE)  que  diu  si  aquella  fila  és  una  "Criatura"  o  una  "Terra".  ●  ✅  Avantatge:  És  la  més  ràpida.  Per  recuperar  una  carta  no  cal  fer  cap  JOIN,  només  llegir  una  fila.  ●  ❌  Inconvenient:  Molt  "bruta".  Si  una  fila  és  una  Terra,  les  columnes  de  força i  resistència estaran  buides  (NULL).  Si  tens  molts  tipus  de  cartes  diferents,  tindràs  una  taula  amb  centenars  de  columnes  plenes  de  buits.  
import  javax.persistence.*;   @Entity  //  Estratègia:  Tot  a  una  taula  @Inheritance(strategy  =  InheritanceType.SINGLE_TABLE)  //  Columna  'chivata'  que  dirà  "CRIATURA"  o  "TERRA"  @DiscriminatorColumn(name  =  "tipus_carta",  discriminatorType  =  DiscriminatorType.STRING )  public  abstract  class  Carta  {      @Id  @GeneratedValue      private  Long  id;      private  String  nom;  }   @Entity

//  Valor  que  es  posarà  a  la  columna  'tipus_carta'  @DiscriminatorValue("CRIATURA")   public  class  Criatura  extends  Carta  {      private  int  forca;      private  int  resistencia;  }  
Resultat  a  la  BD  (SQL):  Taula  CARTA amb  columnes:  id,  nom,  tipus_carta,  forca,  resistencia.  
Nota:  Si  la  carta  és  una  Terra,  els  camps  forca i  resistencia seran  NULL.  
4.2.  
JOINED
 
(Taules
 
unides)
 
Concepte:  Cada  classe  té  la  seva  pròpia  taula.  La  taula  filla  només  guarda  els  seus  camps  i  la  ID  que  la  connecta  amb  el  pare.  
Aquesta  és  la  més  "neta"  i  la  que  més  agrada  als  administradors  de  bases  de  dades  clàssics.  
●  Com  es  guarda:  Es  crea  una  taula  per  a  cada  classe.  ○  Taula  CARTA:  Guarda  id,  nom i  cost.  ○  Taula  CRIATURA:  Guarda  l'id i  la  força.  ○  Taula  TERRA:  Guarda  l'id i  el  color_produccio.  ●  Com  es  recupera:  Si  demanes  una  Criatura,  JPA  ha  de  fer  un  JOIN  entre  la  taula  CARTA i  la  taula  CRIATURA usant  l'ID.  ●  ✅  Avantatge:  No  hi  ha  camps  buits.  Cada  taula  té  només  el  que  li  toca.  Respecta  la  integritat  de  les  dades.  ●  ❌  Inconvenient:  És  més  lenta.  Cada  vegada  que  vols  una  carta  filla,  la  base  de  dades  ha  de  buscar  en  dues  taules  diferents  i  "enganxar-les".  
 
import  javax.persistence.*;   @Entity  //  Estratègia:  Taules  separades  unides  per  ID  @Inheritance(strategy  =  InheritanceType.JOINED)   public  abstract  class  Carta  {      @Id  @GeneratedValue      private  Long  id;      private  String  nom;  }   @Entity  //  Opcional:  Personalitzar  el  nom  de  la  clau  forana  @PrimaryKeyJoinColumn(name  =  "carta_id")   public  class  Criatura  extends  Carta  {      private  int  forca;      private  int  resistencia;  }

●  Taula  CARTA:  id,  nom.  ●  Taula  CRIATURA:  carta_id (PK  i  FK),  forca,  resistencia.  
Nota:  Per  recuperar  una  criatura,  la  BD  ha  de  fer  un  INNER  JOIN.  
4.3  TABLE_PER_CLASS  (Una  taula  per  classe)  
Concepte:  La  classe  pare  no  té  taula  pròpia  (o  no  guarda  dades  dels  fills).  Cada  filla  té  una  taula  completa  amb  TOTS  els  camps  (heretats  i  propis).  
Aquesta  és  la  menys  utilitzada,  però  és  bo  conèixer-la.  
●  Com  es  guarda:  S'ignora  la  classe  pare  com  a  taula.  Es  crea  una  taula  per  a  cada  fill  que  sigui  "instanciable".  ○  Taula  CRIATURA:  Té  id,  nom,  cost,  força.  ○  Taula  TERRA:  Té  id,  nom,  cost,  color_produccio.  ●  ✅  Avantatge:  Si  només  busques  Criatures,  és  molt  ràpid  perquè  tota  la  info  (inclosa  la  del  pare)  està  en  una  sola  taula.  ●  ❌  Inconvenient:  El  gran  problema  és  el  polimorfisme .  Si  fas  una  consulta  tipus  SELECT  c  FROM  Carta  c,  la  base  de  dades  ha  de  fer  una  consulta  a  totes  les  taules  de  fills  i  ajuntar  els  resultats  amb  un  UNION.  És  molt  poc  eficient  per  a  consultes  genèriques.  
import  javax.persistence.*;   @Entity  //  Estratègia:  Cada  filla  té  la  seva  taula  autònoma  @Inheritance(strategy  =  InheritanceType.TABLE_PER_CLASS)   public  abstract  class  Carta  {      @Id  @GeneratedValue(strategy  =  GenerationType.AUTO)  //  Compte:  IDENTITY  sol  fallar  aquí      private  Long  id;      private  String  nom;  }   @Entity  public  class  Criatura  extends  Carta  {      //  A  la  BD  tindrà  columnes:  id,  nom,  forca,  resistencia      private  int  forca;      private  int  resistencia;  }   @Entity  public  class  Terra  extends  Carta  {      //  A  la  BD  tindrà  columnes:  id,  nom,  color      private  String  color;  }  
Resultat  a  la  BD  (SQL):  
●  Taula  CRIATURA:  id,  nom,  forca,  resistencia.

●  Taula  TERRA:  id,  nom,  color.  ●  No  hi  ha  taula  CARTA amb  dades  barrejades.  
Nota:  Si  fas  SELECT  c  FROM  Carta  c,  la  BD  ha  de  fer  un  UNION de  totes  les  taules.  
Resum:  
●  SINGLE_TABLE: Molt  ràpida  (sense  joins).,"Molts  NULLs,  taula  bruta."  ●  JOINED: Dades  netes  i  normalitzades.,Més  lenta  (fa  Joins).  ●  TABLE_PER_CLASS: Fàcil  si  només  consultes  fills.,Lenta  per  consultes  polimòrfiques  (UNION).
